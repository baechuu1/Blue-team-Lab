Attack: SSH Brute Force

1. Overview

A brute force attack attempts to gain unauthorized access by trying multiple password combinations against a target system. This is commonly observed in SSH services exposed to a network.

2. Lab Environment
Attacker Machine: Kali Linux
Target Machine: Ubuntu Server
Monitoring Tool: Splunk Enterprise
Logs Monitored:
/var/log/auth.log
/var/log/syslog
Network Setup: Host-only adapter (isolated lab environment)

3. Attack Execution

From Kali Linux, a brute force attack was conducted using Hydra:

hydra -l <username> -P password.txt ssh://<target-ip>
What Happened:
Multiple failed login attempts were generated
Eventually resulted in a successful login
4. Logs Generated (Ubuntu)

Example findings in /var/log/auth.log:

Key Observations:
High volume of failed login attempts
Same source IP repeatedly attempting access
Eventual successful authentication

5. Detection in Splunk
🔍 Detect Failed Login Attempts
index=linux_logs "Failed password"
| stats count by src_ip, user
| where count > 5
🔍 Detect Successful Login After Failures
index=linux_logs ("Failed password" OR "Accepted password")
| transaction src_ip maxspan=5m
What This Shows:
Identifies brute force behavior
Correlates failed attempts with eventual success

6. Indicators of Compromise (IOCs)
Repeated failed login attempts from a single IP
Login attempts against multiple usernames
Successful login following multiple failures
7. MITRE ATT&CK Mapping
T1110 – Brute Force

8. Remediation / Defense
Implement account lockout policies
Enable Multi-Factor Authentication (MFA)
Restrict SSH access (e.g., allow specific IPs)
Deploy tools like fail2ban
Monitor authentication logs continuously

9. Key Takeaways
Brute force attacks are noisy and detectable through log analysis
Correlating failed and successful logins is critical
SIEM tools like Splunk can quickly identify patterns and anomalies
